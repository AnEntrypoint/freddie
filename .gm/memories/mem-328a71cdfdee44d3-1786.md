## Resolved mutable: windows-cmd-bash-tool-quoting-broken

Precisely reproduced via a real script file (avoiding nested-shell escaping artifacts from testing through Bash itself): spawn('cmd', ['/c', command], {cwd}) with the EXACT command string the model sent (node -e "console.log(JSON.stringify(require('./project4/parse.js').parseCsv('name,age\\nJoe,5')))" -- literal two-char backslash-n preserved) produces exitCode:0, stdout:'', stderr:'' -- a completely silent success with NO output and NO error, on Windows cmd.exe specifically. This is a genuine freddie bash-tool defect: certain nested-quote patterns (double-quoted node -e body containing single-quoted JS strings with an embedded literal backslash-n) cause cmd.exe to silently drop all child process output while still reporting success. This is worse than an error -- it looks like the command ran fine and produced nothing, which is exactly what led the chat model to (wrongly) conclude 'bash tool is not available' and fall back to an unverified from-scratch reimplementation via code_execution instead of actually testing the real file. Root cause is in cmd.exe's argument tokenization of nested quotes with backslash-escaped characters, not in plugins/bash/handler.js's spawn call itself (the spawn is a correct, standard child_process usage) -- this is a known class of cmd.exe quoting fragility for complex one-liners. Scope: real, verified, but a defensive fix (e.g. detecting a suspiciously-empty stdout+stderr+exitCode:0 combination and warning the caller, or writing complex node -e payloads to a temp .js file and running `node tempfile.js` instead of -e to sidestep cmd quoting entirely) is a genuine improvement opportunity, tracked as its own PRD row rather than blocking this turn's advanced-task testing.
