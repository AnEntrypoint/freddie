Freddie persistent-compute layer: src/machines/step-journal.js sits atop snapshot-store to close the recompute window. snapshot-store resumes from the last committed xstate transition; if a process dies while a fromPromise invoke is in flight (LLM HTTP call sent, or tool running) the snapshot is still pre-invoke and a naive resume RE-RUNS that call. runStep(sessionKey,stepId,fn) fixes this: a completed step_results row (libsql, keyed by session_key+step_id) returns its cached JSON result WITHOUT calling fn (at-most-once LLM billing + tool side-effects); a started-only row (crash mid-fn) re-runs fn (at-least-once for exactly the one interrupted call); falsy sessionKey just runs fn. An in-process _inflight Map locks concurrent same-key calls. Wiring: agent machine.js prompting invoke wraps llm() in runStep(sessionKey,llm:<iteration>); executing_tools wraps each call (preToolCall+dispatchTool+postToolCall, returning {content,extras}) in runStep(sessionKey,tool:<iteration>:<tcid>) so hooks do not re-fire on resume; driveAgentActor clearSteps on done. batch.js wraps runOne in runStep(batchId,prompt:<i>); cron tick wraps fire in runStep(cron:<jobid>,fire:<minuteKey>); gateway handleInbound wraps runTurn in runStep(msgKey,run); acp prompt.submit journals the turn (sessionKey=acp:<sid>) plus an acp-persist step. stepId keys use the persisted iteration counter so they are stable across resume. GET /api/machines/steps/:key surfaces the journal. test.js step-journal-compute group + node -e probe (1 LLM invocation across 3 re-entries) witness at-most-once.
